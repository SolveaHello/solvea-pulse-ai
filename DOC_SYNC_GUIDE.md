# Pulse AI — AI 驱动的文档同步机制教程

> **适用对象**：产品经理、前后端工程师、技术负责人
> **文档目的**：说明本项目如何通过 AI + 自动化脚本，保证需求文档、设计文档与实现代码的长期一致性

---

## 一、我们在解决什么问题

### 传统产研协作的文档困境

几乎所有团队都经历过这种痛苦：

```
产品更新了需求文档
    ↓
通知了工程师（或者没有）
    ↓
工程师按自己的理解改了代码
    ↓
技术文档没人更新
    ↓
三个月后，没有人知道代码和文档哪个是对的
```

**结果**：文档成为摆设，新人上手靠口耳相传，改需求全靠"我记得当时说的是……"

### 根本原因

问题不是"大家不认真"，而是**文档维护的成本太高、反馈太慢**：

- 改一个流程，要同步修改 PRD、前端文档、后端文档、甚至测试文档
- 人工对比"哪里不一致"需要大量时间，往往被跳过
- 没有强制机制，文档同步完全依赖个人自律

### 本项目的解法

**用 AI 承担"文档一致性检查员"的角色**，将文档维护成本从"小时级"压缩到"分钟级"，从"人工对比"变成"自动提示 + AI 执行"。

## 二、文档体系设计

### 单一事实来源（Single Source of Truth）

本项目的所有需求与设计，最终来源只有一份：

```
MARKETING_FLOW.md
```

其他三份文档是它在不同维度的"投影"：

```
MARKETING_FLOW.md（流程 + 业务逻辑层）
       │
       ├─► PRD.md
       │   视角：产品经理
       │   内容：用户故事、功能优先级、数据模型、API 规范、成功指标
       │
       ├─► FRONTEND_FLOW.md
       │   视角：前端工程师
       │   内容：路由设计、组件结构、状态管理、页面交互、环境变量
       │
       └─► BACKEND_FLOW.md
           视角：后端工程师
           内容：API 路由表、数据模型、服务模块、Webhook、定时任务
```

**这决定了变更的方向是单向的**：

```
✅ 正确：先改 MARKETING_FLOW.md → 再同步其他三份
❌ 错误：直接改 PRD.md 或 FRONTEND_FLOW.md → 与 FLOW 产生分歧
```

### 文档职责边界

| 文档 | 负责回答 | 不负责回答 |
|------|---------|----------|
| MARKETING_FLOW.md | 为什么有这个功能？完整流程是什么？ | 代码怎么写？用哪个组件？ |
| PRD.md | 要做什么？验收标准是什么？ | 具体怎么实现？ |
| FRONTEND_FLOW.md | 前端页面怎么组织？状态怎么管理？ | 接口数据结构？业务逻辑？ |
| BACKEND_FLOW.md | 接口如何设计？数据如何存储？ | 前端怎么展示？产品为什么这样设计？ |

## 三、三层同步机制详解

### Layer 1 — CLAUDE.md：规范层

**文件位置**：`/CLAUDE.md`（项目根目录）

**工作原理**：

Claude Code（AI 编程助手）在每次会话开始时，会自动读取项目根目录下的 `CLAUDE.md` 文件，将其内容作为工作规范加载到上下文中。

这意味着：**你不需要每次都提醒 AI 去同步文档**，规范已经内置在 AI 的工作流程里。

```
你打开 Claude Code 开始工作
         ↓
Claude 自动读取 CLAUDE.md
         ↓
Claude 知道：
  "修改 MARKETING_FLOW.md 后必须同步哪些文档"
  "修改代码后必须更新 Phase 进度表"
  "哪份文档是 Single Source of Truth"
         ↓
每次你修改 FLOW 文档，Claude 会主动提示同步
```

**CLAUDE.md 定义的核心规则**：

```markdown
每次修改 MARKETING_FLOW.md 后，必须检查并同步：

| 变更类型              | 需更新文档              |
|---------------------|----------------------|
| 新增/删除流程步骤      | PRD + FRONTEND + BACKEND |
| 修改页面功能或UI设计   | FRONTEND             |
| 新增/修改API接口       | BACKEND + PRD        |
| 修改数据字段           | BACKEND + PRD        |
| 修改Disposition/状态机 | PRD + BACKEND + FRONTEND |
| Phase状态变更          | PRD + FLOW           |
```

**效果**：将文档同步规范从"靠记忆"变成"AI 强制执行"。

---

### Layer 2 — 影响域映射表：定位层

**位置**：`MARKETING_FLOW.md` 头部的注释表格

**工作原理**：

`MARKETING_FLOW.md` 每个章节头部，都标注了"如果我变了，你需要去更新哪些文档的哪些章节"。这张表是一个精确的**变更传播图**。

```markdown
| 本文档章节              | 影响 PRD.md     | 影响 FRONTEND_FLOW.md        | 影响 BACKEND_FLOW.md     |
|------------------------|----------------|------------------------------|--------------------------|
| LINE A STEP 3（外呼）  | §四4.3-4.4     | hooks/useCallStatus          | services/vapi            |
| LINE A STEP 4（详情页）| §七7.1页面表   | routes/CampaignDetailPage    | routers/campaigns GET    |
| Disposition 状态机     | §三状态机      | types/index.ts               | models/contact enum      |
```

**解决的问题**：

以前改一个功能，工程师需要阅读整份 PRD 才能确认哪里受影响。有了这张映射表，**从"全文搜索"变成"精准定位"**，大幅降低同步成本。

---

### Layer 3 — sync-docs.sh：执行层

**文件位置**：`/scripts/sync-docs.sh`

**工作原理**：

脚本通过 `shasum` 对 `MARKETING_FLOW.md` 做哈希快照。每次运行时，对比当前文件与上次快照的差异，分析变更区域的关键词，判断哪些文档需要同步，最终输出精准的执行 prompt。

```bash
bash scripts/sync-docs.sh
```

**执行流程**：

```
运行脚本
    ↓
计算 MARKETING_FLOW.md 当前 hash
    ↓
与上次保存的快照对比
    ↓
[无变更] → 输出"文档状态一致"，退出
    ↓
[有变更] → diff 分析变更内容
    ↓
关键词匹配（STEP / API / 页面 / 数据模型 / RFM...）
    ↓
输出受影响文档列表 + 精准章节定位
    ↓
生成可直接使用的 Claude 同步 prompt
    ↓
你把 prompt 发给 Claude Code 执行同步
    ↓
同步完成后按 Enter → 更新快照
```

**关键词分类逻辑**：

| 关键词类型 | 触发同步 |
|-----------|---------|
| `STEP` / `Phase` / `状态机` / `双线` | PRD.md |
| `页面` / `UI` / `组件` / `路由` / `看板` | FRONTEND_FLOW.md |
| `API` / `数据模型` / `字段` / `Webhook` / `Service` | BACKEND_FLOW.md |

## 四、完整工作流演示

### 场景：产品决定在 Campaign 详情页新增"批量重拨"功能

#### 第一步：修改 MARKETING_FLOW.md

```markdown
# 在 STEP 4 通话记录列表中新增：

## 批量操作
- 勾选多条 NO_ANSWER 记录 → 点击「批量重拨」
- 系统将选中联系人重新加入外呼队列
- 状态重置为 PENDING，下一个执行批次自动拨出
```

#### 第二步：运行同步检查脚本

```bash
bash scripts/sync-docs.sh
```

输出：

```
⚠️  检测到 MARKETING_FLOW.md 已变更

📋 变更影响域分析：

  → PRD.md  需要同步
     章节：§四4.3 AI外呼模块、§六6.2 Campaign API

  → FRONTEND_FLOW.md  需要同步
     章节：routes/CampaignDetailPage 组件设计

  → BACKEND_FLOW.md  需要同步
     章节：API路由表、routers/campaigns

─────────────────────────────────────────────────────
📝 请将以下 prompt 发送给 Claude Code 执行同步：

请根据最新的 MARKETING_FLOW.md（STEP 4 新增批量重拨功能），
同步更新以下文档：
- PRD.md（§四4.3功能列表新增批量重拨，§六新增 POST /campaigns/:id/bulk-redial 接口）
- FRONTEND_FLOW.md（CampaignDetailPage 新增批量勾选与操作栏描述）
- BACKEND_FLOW.md（API路由表新增接口，说明重置 PENDING 逻辑）

请逐文档更新，完成后输出变更摘要。
```

#### 第三步：把 prompt 交给 Claude Code

Claude 读取所有文档后，精确更新三份文档的对应章节，完成后报告：

```
✅ PRD.md — 在 §四4.3 功能列表新增「批量重拨」P1 条目，
            在 §六 新增 POST /campaigns/:id/bulk-redial 接口定义

✅ FRONTEND_FLOW.md — 在 CampaignDetailPage 通话记录列表描述中，
                      新增多选操作栏和批量重拨按钮说明

✅ BACKEND_FLOW.md — 在 campaigns 路由表新增接口，
                     在业务逻辑中说明批量重置 PENDING 的处理流程
```

#### 第四步：确认同步，更新快照

```bash
同步完成后，按 Enter 更新快照（标记为已同步）...
✅ 快照已更新。文档状态：已同步
```

**总耗时：约 5 分钟**（传统方式手动对比更新：30-60 分钟）

## 五、新生产方式的优势

### 对比传统方式

| 维度 | 传统方式 | 本机制 |
|------|---------|-------|
| **文档同步触发** | 靠人记、靠提醒 | 脚本自动检测变更 |
| **影响范围判断** | 人工阅读全文对比 | 关键词映射，精准定位章节 |
| **同步执行** | 人工逐份修改，易遗漏 | AI 批量执行，逐份报告 |
| **同步耗时** | 30-60 分钟/次 | 3-5 分钟/次 |
| **漏同步风险** | 高（完全依赖人工） | 低（hash 快照记录变更，未同步有提示） |
| **新人上手** | 靠口耳相传 | CLAUDE.md 规范自动加载 |
| **文档可信度** | 逐渐降低，最终成摆设 | 持续保鲜，始终可信 |

### 对产研团队工作方式的改变

#### 对产品经理

**过去**：写好需求文档后，用飞书/Slack 通知工程师，靠沟通保证理解一致。需求变更时，自己手动更新多份文档，经常漏改。

**现在**：只需维护 `MARKETING_FLOW.md` 一份文档，AI 自动将变更传播到所有技术文档。**产品经理从"多文档维护者"变成"单一来源的守护者"**。

```
Before: PM 维护 PRD + 口头同步技术文档（经常脱节）
After:  PM 只维护 MARKETING_FLOW.md → 自动同步所有文档
```

#### 对前端工程师

**过去**：拿到需求后，需要同时阅读 PRD 和口头沟通，才能理解完整意图，经常出现"做完了但方向不对"的返工。

**现在**：`FRONTEND_FLOW.md` 始终与最新需求同步，每个页面的功能点、组件职责、状态管理都有清晰描述，**像在读一份实时更新的实现说明书**。

```
Before: 工程师自行理解 PRD → 实现 → 返工
After:  FRONTEND_FLOW.md 精确描述实现细节 → 直接参照开发
```

#### 对后端工程师

**过去**：API 接口设计靠口头约定，联调阶段才发现前后端对字段的理解不一致，修改成本高。

**现在**：`BACKEND_FLOW.md` 随需求同步更新，**接口变更有完整的 diff 记录**，联调前双方查同一份文档即可对齐。

#### 对技术负责人

**过去**：每次 Code Review 除了看代码，还要检查文档是否同步更新，是额外负担。

**现在**：文档同步由机制保障，Code Review 可专注于代码质量本身。**技术债不再从"文档腐烂"开始积累**。

## 六、机制设计原则

### 原则一：最小阻力路径

好的规范必须比不遵守它更简单。本机制的设计目标是：

> 维护文档一致性的成本，要低于维护不一致所带来的沟通成本

`sync-docs.sh` 的设计：一行命令，30 秒得到结果，直接给出可执行的 prompt —— **让"做对的事"比"做错的事"更省力**。

### 原则二：AI 做苦力，人做决策

文档同步是重复性的体力劳动，最适合 AI 执行。人的精力应该花在：

- 判断需求是否合理
- 验收 AI 更新的结果是否准确
- 处理需要业务背景才能判断的边界情况

**AI 负责执行，人负责审核**，而不是反过来。

### 原则三：变更有迹可查

每次同步后脚本都会更新哈希快照，`.doc-snapshots/` 目录保存了每次的文档备份。这意味着：

- 任何时候都能看到"上次同步后改了什么"
- 出现争议时，可以追溯需求变更的完整历史
- **文档的版本历史，就是产品决策的历史**

## 七、日常使用 SOP

### 需求变更时

```
1. 修改 MARKETING_FLOW.md
2. 运行：bash scripts/sync-docs.sh
3. 复制输出的 prompt，发给 Claude Code
4. 审核 Claude 更新的三份文档
5. 确认无误，按 Enter 更新快照
```

### 开发新功能时

```
1. 查阅 FRONTEND_FLOW.md / BACKEND_FLOW.md（已与需求同步）
2. 开发完成后，更新代码
3. 告知 Claude Code：「功能 X 已完成，更新 Phase 进度表」
4. Claude 自动将 Phase 表中的 ⬜ 改为 ✅
```

### 新人入职时

```
1. 阅读 MARKETING_FLOW.md（理解产品全貌和业务逻辑）
2. 阅读对应角色的实现文档（FRONTEND 或 BACKEND）
3. 无需找人口头同步，文档即是最新状态
```

### 每次 Sprint 结束时

```
1. 运行 sync-docs.sh 确认所有文档已同步
2. 更新 MARKETING_FLOW.md 中的 Phase 进度表
3. 同步传播到 PRD.md 和实现文档
```

## 九、这套机制背后的思维转变

传统产研协作的假设是：

> "我们靠流程和自律来保证文档一致性"

这套机制的假设是：

> "人不应该为重复性的一致性检查负责，AI 可以更好地完成这件事"

这不是技术上的创新，而是**生产关系的重新分工**：

```
旧分工：人 写文档 + 人 同步文档 + 人 检查一致性
新分工：人 写关键决策 + AI 执行同步 + 人 审核结果
```

当 AI 承担了文档维护的机械性工作，**人的创造力可以完全专注在"做什么"和"为什么"上，而不是"更新哪份文档的哪个章节"**。

这是 AI 工具真正改变生产方式的地方：不是替代人的判断，而是**消除人不应该花时间的摩擦**。
